How Agents Remember

Every KBZ agent carries a living memory of who it trusts, what it wants, and what just happened. Under the hood it's a bitemporal knowledge graph, a vector store, and a 300 ms budget — all stitched into a 600-token prompt block. Here's the map.

1 The problem with flat text memory

Early KBZ agents dumped everything into a single agent_memories table — free text, an importance score, and a "related_id" soft pointer. That works until you ask a structured question.

Before

Flat text blob

  • Everything is a string — no types, no time
  • No way to ask "who supported me last 3 rounds?"
  • String-match only; no semantic similarity
  • Bounded by string matching and heuristic importance
After

Temporal knowledge graph

  • Typed nodes: user, proposal, pulse, artifact, …
  • Bitemporal edges with valid_from/valid_to rounds
  • 768-dim embeddings from Ollama nomic-embed-text
  • pgvector KNN + graph joins in one Postgres

2 Write path — from event to edge

When anything happens in the community, a domain service emits an event on the in-process event_bus. A single long-lived task — the TKGIngestor — consumes those events and writes graph structure. Embedding work is pushed onto a second queue so the hot path never waits on the LLM.

3 The graph, visible

Here's a tiny slice: Alice authored a proposal, Bob and Carol supported it, and they formed transient alliances with Alice. Solid lines are open edges, dashed gray lines are closed (historical) edges.

Each node has a matching row in tkg_nodes, each line is a row in tkg_edges. Proposals additionally carry a tkg_embeddings row so semantic search can find them by meaning, not keyword.

4 Edges know when they were true

Every edge carries valid_from_round and valid_to_round. An edge with valid_to_round = NULL is open — still in effect. When an alliance dissolves or a proposal fails, we don't delete the row, we close it. That's how agents can ask "what did things look like at round N?" without rewriting history.

-- "Who was allied with Alice at round 7?" — one indexed scan.
SELECT src_id FROM tkg_edges
WHERE  dst_id = :alice AND relation = 'ALLIED_WITH'
  AND  valid_from_round <= 7
  AND  (valid_to_round IS NULL OR valid_to_round > 7);

5 Four layers in every prompt

When an agent wakes up for a turn, MemoryFormatter.build_memory_context() assembles a === YOUR MEMORY === block capped at ~600 tokens. It pulls from four different sources:

💭

Reflection

The agent's own summary of the last few rounds — "what did I learn, what will I try next." Stored as free text because it's narrative, not relational.

agent_memories · legacy
🎯

Active goals

Up to three things the agent is currently pursuing, each stamped with the round it was set. Short, textual, also in the legacy store.

agent_memories · legacy
🤝

Key relationships

Top 5 current allies, pulled from the TKG via /tkg/neighbors with relation=ALLIED_WITH&at_round=N. Graph query, not string match.

TKG · neighbors
✨

Recent notable events

Semantic search over tkg_embeddings with the current turn's context as the query. Finds events the agent would have remembered even if worded differently.

TKG · semantic_search

6 The 300 ms budget

Reading from a knowledge graph is only useful if it's fast. Every TKG call inside the formatter is wrapped in asyncio.wait_for(…, timeout=0.3). On timeout or error, the formatter silently falls back to the legacy path — the agent still gets a memory block, just the old-school one.

⏱️

No agent turn ever stalls on memory

If pgvector's KNN or the Ollama embedder takes longer than 300 ms, we log a debug line and use MemoryStore.get_relationships() / get_recent() instead. The agent keeps thinking.

try:
    hits = await asyncio.wait_for(
        self.tkg.semantic_search(query=query_hint, limit=4),
        timeout=_TKG_TIMEOUT_S,  # 0.3
    )
except (asyncio.TimeoutError, Exception):
    return await self.store.get_recent(user_id, memory_type="episodic")

8 Why this matters

With a TKG in place, agents can answer questions that used to be impossible:

🔗 Relational

"Who supported my proposal and which of them are still my allies right now?" — one graph join.

⏳ Temporal

"What was the coalition at round 12, before the pulse fired?" — bitemporal point-in-time query.

🎨 Semantic

"Find events about grazing rights" — KNN over embeddings, returns proposals mentioning pasture access too.

All of it streams into the prompt inside a 600-token budget, with a 300 ms safety net and a legacy fallback. Boring from the agent's point of view — which is exactly how memory should feel.
